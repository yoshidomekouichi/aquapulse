### **データフロー比較**

#### **現在（ラズパイ）**

```python
# collector/src/main.py

while True:
    # 1. センサー読み取り
    readings = []
    for source in SOURCES:
        data = source.get_readings()
        readings.extend(data)
    
    # 2. DB書き込み（直接）
    save_readings(readings)
    
    time.sleep(60)

問題:
  - エラーでデータ消失
  - スレッド管理が複雑
  - スケールしない
```

#### **移行後（ESP32 + GCP）**

```python
# esp32/main.py

while True:
    # 1. センサー読み取り
    temp = ds_sensor.read_temp(rom)
    
    # 2. MQTT送信（バッファリング）
    mqtt.publish("aquapulse/sensor", {"value": temp})
    
    time.sleep(60)

# cloud-functions/ingest/main.py

def ingest(event, context):
    # 3. Pub/Subから受信
    data = decode(event['data'])
    
    # 4. BigQuery保存
    client.insert_rows_json(table, [data])

メリット:
  ✅ データ保証（Pub/Sub）
  ✅ シンプル（各50行）
  ✅ スケーラブル
```

## 🛡️ **リスク管理**

### **想定リスクと対策**

| リスク | 影響度 | 対策 |
|--------|--------|------|
| **ESP32が壊れる** | 中 | 予備機を1台購入（$15） |
| **WiFi接続が切れる** | 中 | Pub/Subでバッファリング（7日間） |
| **GCPコストが増える** | 低 | Cloud Billingアラート設定 |
| **データ欠損** | 中 | Dev環境で十分テスト |
| **Grafana接続エラー** | 低 | BigQueryは生きてる、復旧可能 |

### **ロールバックプラン**

```
問題発生時:

1. ESP32停止
2. ラズパイ再起動
3. 旧システムで稼働継続
4. 原因調査
5. 修正後、再度移行
```
