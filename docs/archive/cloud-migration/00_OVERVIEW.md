**AquaPulse ESP32 + GCP 構成の全体像**

## 🚨 **移行の背景と決断理由**

### **現在直面している問題**

```
🔴 緊急事態: ラズパイが完全に操作不能

症状:
  ❌ SSH接続が完全に不通（Mac → ラズパイ）
  ❌ Tailscaleも接続できず
  ❌ IP固定していたが接続不可
  ❌ DHCP予約も未設定
  ❌ 物理アクセス不可（水槽の裏、モニター/キーボード未接続）

結論: OSリセット不可避
```

### **なぜラズパイに戻さないのか**

#### **再発リスクが高い**

```
過去の試行錯誤:
  ✅ IP固定 → 解決せず
  ✅ Tailscale導入 → 解決せず
  ✅ Cloud Agentで調査 → 原因特定できず

→ ネットワーク設定の問題ではなく、
  物理配置（水槽の裏）やハードウェア不安定性が根本原因の可能性
```

#### **コスト比較（復旧 vs 移行）**

| 作業内容 | ラズパイ復旧 | クラウド移行 | 備考 |
|---------|------------|------------|------|
| **OSリセット** | 必要 | 不要 | SDカード抜き差し困難 |
| **環境再構築** | 必要（Docker等） | 必要（GCP等） | 手間はほぼ同等 |
| **`.env`再設定** | 必要 | 必要 | 同じ |
| **動作テスト** | 必要 | 必要 | 同じ |
| **将来の再発** | **高リスク** | **ゼロ** | ← ここが決定的 |

**結論: 手間が同じなら、根本解決できる方を選ぶべき**

### **今が移行の最適タイミングである理由**

```
✅ どうせゼロから構築 = 移行障壁が最低
✅ データ喪失は避けられない = 既存環境への未練なし
✅ ラズパイ物理アクセス困難 = OSリセット作業自体が高リスク
✅ 同じ苦労をするなら将来性の高い構成へ
```

**このガイドは「ラズパイが死んだ」状況からの脱出プランです。**

### **データフロー比較**

#### **現在（ラズパイ）**

```python
# collector/src/main.py

while True:
    # 1. センサー読み取り
    readings = []
    for source in SOURCES:
        data = source.get_readings()
        readings.extend(data)
    
    # 2. DB書き込み（直接）
    save_readings(readings)
    
    time.sleep(60)

問題:
  - エラーでデータ消失
  - スレッド管理が複雑
  - スケールしない
```

#### **移行後（ESP32 + GCP）**

```python
# esp32/main.py

while True:
    # 1. センサー読み取り
    temp = ds_sensor.read_temp(rom)
    
    # 2. MQTT送信（バッファリング）
    mqtt.publish("aquapulse/sensor", {"value": temp})
    
    time.sleep(60)

# cloud-functions/ingest/main.py

def ingest(event, context):
    # 3. Pub/Subから受信
    data = decode(event['data'])
    
    # 4. BigQuery保存
    client.insert_rows_json(table, [data])

メリット:
  ✅ データ保証（Pub/Sub）
  ✅ シンプル（各50行）
  ✅ スケーラブル
```

## 🛡️ **リスク管理**

### **想定リスクと対策**

| リスク | 影響度 | 対策 |
|--------|--------|------|
| **ESP32が壊れる** | 中 | 予備機を1台購入（$15） |
| **WiFi接続が切れる** | 中 | Pub/Subでバッファリング（7日間） |
| **GCPコストが増える** | 低 | Cloud Billingアラート設定 |
| **データ欠損** | 中 | Dev環境で十分テスト、Phase 1で長期稼働確認 |
| **Grafana接続エラー** | 低 | BigQueryは生きてる、復旧可能 |
| **センサー値が異常** | 中 | 想定範囲外ならセンサー配線チェック |

### **ロールバックプラン（制約あり）**

```
⚠️  重要: ラズパイが動かないため、従来型のロールバックは不可能

問題発生時の対応:

1. ESP32停止（一旦センサー読み取り停止）
2. 原因調査
   - Cloud Logging でエラー確認
   - BigQueryでデータ確認
   - ESP32シリアルモニタでログ確認
3. 修正
   - コード修正 → GitHub push → 自動デプロイ
   - またはgcloud CLIで手動ロールバック
4. 再起動・検証

ラズパイ復旧は選択肢にない:
  ❌ SSH接続不可
  ❌ 物理アクセス困難
  → 新システムで問題解決するしかない
```

### **最悪のシナリオと対策**

```
最悪: ESP32 + GCP も動かない場合

短期対策:
  - 水槽の基本管理は手動に戻す
  - 水温計、TDSメーターを手動で使う
  
中期対策:
  - ESP32を一から作り直す（ハードウェアは安価）
  - GCPは生きているので、ESP32側だけ修正
  
長期対策:
  - トラブルシューティングガイド充実
  - コミュニティ（ESP32フォーラム等）に質問
```
