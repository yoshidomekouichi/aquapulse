# TUI バグ修正の技術説明（2026-03）

## 質問1: なぜ前のグラフが消えずにダブっていたのか？ そして、それをどう修正するか？

**原因**: plotext はグローバルなシングルトン `_figure` を使用する。4つの GraphPanel が順次 `build_pipboy_line_plot()` を呼ぶ際、各呼び出しで `clf()` を呼ぶが、**前のパネルの build() 完了後、次のパネルの clf() が呼ばれる前に、別スレッドで refresh が走る**など、タイミングによっては plotext の内部状態が混在する可能性がある。また、`clear_data()` と `clear_figure()` の使い分けが不十分で、描画設定（axes, grid 等）が残り、次の plot の描画と重なることがある。

**修正**: 各 `build_pipboy_line_plot()` の**先頭**で `plt.clear_figure()` を必ず呼び、**最後**に `try/finally` で確実に `plt.clear_figure()` を再度呼ぶ。これにより、次のパネル呼び出し時には完全にクリーンな状態になる。

## 質問2: なぜグラフが左に寄ってしまったのか？ Textualのパネルサイズいっぱいに広げるために、どのメソッドを修正するか？

**原因**: `build_pipboy_line_plot()` に固定値 `PLOT_WIDTH=42`, `PLOT_HEIGHT=16` を渡していた。6x12 フォントで 800x480 の画面では、1パネルあたり約 65 桁 × 40 行相当のマス目が使えるが、42×16 の固定サイズのためグラフが左に寄って小さく表示されていた。

**修正**: `GraphPanel.update_data()` 内で `#graph` Static ウィジェットの `size.width` と `size.height` を取得し、`build_pipboy_line_plot()` に渡す。これにより、Textual のレイアウトで割り当てられた実際の描画領域いっぱいに plotext が描画する。サイズが 0 の場合はフォールバック値を使用。

## 質問3: 今回こそ確実に「X軸の時刻目盛り（Tick）」を表示させるために、どのようなアプローチをとるか？

**原因**: 1) `height_xlabels` が正しく設定されない、2) `xaxes(True, False)` で上軸を非表示にした際、下軸のラベル用スペースが不足する、3) `xticks()` の位置・ラベルが plotext の内部で正しく解釈されていない可能性。

**修正**: 1) `plt.xticks()` を `plt.plot()` の**後**に呼び、`xside="lower"` を明示。2) 描画高さを十分確保（動的サイズで対応）。3) 目盛り位置を `[0, n//4, n//2, 3*n//4, n-1]` のように均等にし、ラベルを `[t0, t1, t2, t3, t4]` の時刻文字列で渡す。4) plotext の `xfrequency()` を 0 に設定して自動目盛りを無効化し、手動 xticks のみを確実に使う。

## 質問4: これまでの「Pip-boyグリーンへの統一」「太いブロックではなくBraille（点字）マーカーを使った極細の折れ線にする」という要件は、今回の修正でも完全に維持されているか？

**はい、維持する。** 変更なし:
- `marker="braille"`, `fillx=False` を維持
- 出力の ANSI 除去後、Pip-boy グリーン（`PIPBOY_GREEN`, `PIPBOY_GRID`）でスタイル適用
- 黒背景・ヘッダー・スクロールバー非表示も維持
