# AquaPulse Rules

このディレクトリは、AIエージェントとの開発におけるルールを管理します。

## ルール構造

```
.cursor/rules/
├─ README.md                # このファイル
├─ 00-base.mdc              # プロジェクト基盤
├─ 10-collaboration.mdc     # 人間とAIの協働
└─ 20-version-control.mdc   # Git/PR戦略
```

## ロード順序

数値プレフィックスでロード順序を制御:
- `00-` が最初（基盤）
- `20-` が最後（詳細）
- 後のファイルが優先（競合時）

## 各ファイルの役割

### 00-base.mdc（プロジェクト基盤）

```
内容:
  - プロジェクト概要（AquaPulse: 淡水水槽モニタリング）
  - 技術スタック（ESP32, GCP, Terraform）
  - ハードウェア制約（センサー固定、USB物理接続必須）
  - コード規約（Python: Black + isort）
  - 絶対に守るべきルール

適用: すべてのセッション
更新頻度: 稀（プロジェクト構造変更時のみ）
```

### 10-collaboration.mdc（人間とAIの協働）

```
内容:
  1. 対話的開発
     - 説明してから実装
     - 選択肢を提示
     - 意思決定ゲート（3つ）
  
  2. コミュニケーション
     - 確証バイアスの回避
     - フラットな評価
     - 技術的正確性を優先
  
  3. エラー透明性
     - Level 1-3 エラー報告
     - 重要操作前の確認
     - 試行錯誤の透明性

適用: すべてのセッション
更新頻度: 低（スタンス変更時のみ）
```

### 20-version-control.mdc（Git/PR戦略）

```
内容:
  1. Git ワークフロー（GitHub Flow）
     - ブランチ戦略
     - コミット規約
     - 禁止事項
  
  2. PR戦略
     - PRサイズ制約（200-400行）
     - 垂直スライス
     - 1 PR = 1論理的変更
  
  3. 依存関係の判断
     - 技術的依存 vs 論理的依存
     - スタックPR vs 並列PR

適用: Git/PR操作時
更新頻度: 低（ワークフロー変更時のみ）
```

## ルール管理の原則

### 1. 適切なサイズを保つ

```
推奨:
  - 各ファイル: 100-500行
  - 合計: 2,000トークン以下

超過時:
  - サブディレクトリに分割
  - 詳細は別ファイルに参照
```

### 2. 重複を避ける

```
NG: 同じ内容を複数ファイルに記載
OK: 1箇所に記載、他は参照
```

### 3. 具体的に書く

```
NG: 「良いコードを書く」
OK: 「関数は50行以下、1つの責務のみ」
```

### 4. バランスを保つ

```
過度なルール化を避ける:
  - すべてを規定しない
  - 創造性の余地を残す
  - 判断を信頼する
```

## 新しいルールの追加

### ステップ1: 分類

```
どのレイヤーに属するか判断:
  - 基盤（00-）: プロジェクト固有の制約
  - コア（10-20）: 開発スタンス、コミュニケーション
  - プロセス（30-40）: Git、PR、デプロイ
  - ドメイン（50-60）: 技術固有（ESP32、GCP等）
```

### ステップ2: 既存ファイルに追加 or 新規作成

```
判断基準:
  - 既存ファイルに関連性がある → 追加
  - 新しい領域 → 新規作成
  - ファイルが500行超過 → 分割検討
```

### ステップ3: PR作成

```
必須事項:
  - 変更理由
  - 影響範囲
  - 使用例
```

## トラブルシューティング

### Q: ルールが適用されない

```
確認事項:
  1. YAML frontmatter が正しいか
  2. alwaysApply: true になっているか
  3. metadata.environments: cloud が設定されているか
```

### Q: ルールが競合している

```
解決策:
  1. 数値プレフィックスを確認（後が優先）
  2. より具体的なルールを後に配置
  3. 重複を削除
```

### Q: ファイルが多すぎる

```
推奨数: 5-8ファイル

超過時:
  - 関連ルールを統合
  - 使用頻度の低いルールを削除
  - サブディレクトリに整理
```

## 競合解決戦略

```
複数のルールが競合する場合:

1. 後のファイルが優先
   00-base.mdc < 40-pr-strategy.mdc

2. より具体的なルールが優先
   全般的ルール < ドメイン固有ルール

3. セキュリティ優先（deny-wins）
   許可ルール < 禁止ルール
```

## 定期レビュー

```
月次レビュー:
  - 使われていないルールを削除
  - 頻繁に違反されるルールを見直し
  - 新しいパターンをルール化

四半期レビュー:
  - ファイル構造の見直し
  - サイズの最適化
  - 重複の削除
```
